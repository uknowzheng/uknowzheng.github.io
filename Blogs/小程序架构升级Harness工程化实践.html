1. Harness Engineering 介绍

1.1 什么是 Harness Engineering

Harness Engineering（Harness 驾驭工程）是构建 AI Agent 生产级基础设施的学科。如果说模型是引擎，Harness 就是操作系统——它管理上下文、编排工具、验证输出、持久化状态，决定 Agent 在真实环境中如何可靠运行。

核心隐喻：

  • 大模型 = CPU，提供底层算力，但不知道要做什么
  • Harness = 操作系统，管理上下文、编排工具、验证输出、持久化状态

行业洞察：

“The model is commodity. The harness is moat.”（模型是商品，Harness 才是护城河）

两个团队使用相同模型，任务完成率可能相差 40 个百分点（60% vs 98%），差异完全来自 Harness 质量。

1.2 Harness 六大核心组件

mindmap
  root((Agent Harness))
    上下文工程
    工具编排
    状态管理
    验证安全
    人机协同
    生命周期管理

上下文工程：决定模型在每个执行步骤看到什么信息。 工具编排：管理模型在每个步骤可以访问哪些工具。 状态管理：持久化 Agent 的工作状态，支持崩溃恢复。 验证安全：在输出到达真实世界前进行检查。 人机协同：对高风险操作设置审批门控。 生命周期管理（Mutil Agent）：管理 Agent 的启动、监控、关闭和恢复。

1.3 提示词工程 vs 上下文工程 vs Harness Engineering

层级范围关注点示例
提示词工程最窄如何写 system prompt 和 user prompt设计指令模板、few-shot 示例
上下文工程中等模型在每个步骤看到什么信息检索文档、历史摘要、工具结果、环境状态
Harness Engineering最广Agent 的完整运行时基础设施上下文管理 + 工具编排 + 验证 + 状态 + 安全

关系总结：

  • 提示词工程是上下文工程的子集
  • 上下文工程是 Harness Engineering 的六大核心组件之一
  • Harness Engineering 是最顶层的工程学科

2.3 状态管理（记忆管理）

任务进度管理

短任务：SDD（基于Spec驱动开发）

对于短平快的任务，使用 OpenSpec 格式定义需求：

OpenSpec三阶段工作流工作流：

flowchart LR
    A[init] --> B[proposal]
    B --> C[apply]
    C --> D[archive]
    D --> B
    
    style A fill:#e1f5fe

特点：

  • 归档一些短期小需求，生成一个 spec snapshot
  • 类似于DB Migrate归档方式
  • 便于复盘和知识复用
  • 可追溯、可回溯

归档方式（类似数据库 Migrate）：

specs/
├── history/              # 历史规格归档
│   ├── 2026-04-15-login/
│   │   ├── open-spec.yaml
│   │   └── accepted.md
│   └── 2026-04-14-user-profile/
│       ├── open-spec.yaml
│       └── accepted.md
└── active/            # 当前活跃任务
    └── open-spec.yaml

2.4 验证安全

Lint&Prettier静态检查

graph LR
    A[增量提交代码] --> B{ESLint + Prettier}
    B -->|自动修复| C[语法检查]
    C -->|语法通过| D[提交通过]
    C -->|语法失败| F[❌ 阻止提交]
    
    style B fill:#fff3e0
    style D fill:#e1f5fe
    style F fill:#ffcdd2

验证层级：

  1. 语法检查：ESLint + Prettier 自动修复
  2. 单元测试：Jest 测试覆盖
  3. 集成测试：E2E 测试验证

TDD （测试驱动开发）

通过单元测试来保障功能正常

flowchart LR
    A[Red: 编写失败测试] --> B[Green: 写最小实现]
    B --> C[Refactor: 重构]
    C --> D{测试通过?}
    D -->|是| E[提交]
    D -->|否| A
    
    style A fill:#ffcdd2
    style D fill:#c8e6c9

核心原则：

  • 基于Vue3的Compoable纯函数进行单元测试用例编写
  • 功能单一&原子化

2.5 人机协同

通过Rule规则进行人工确认：

  • 高风险文件（package.json、路由配置）改动需要人工确认
  • 删除操作需要二次确认

2.6 生命周期管理

flowchart LR
    A[需求确认] --> B[规格定义]
    B --> C{任务拆分&编排}
    C --> D[页面&逻辑开发]
    C --> E[单元测试用例编写]
    D --> F{基于Lint验证}
    E --> F{基于Lint验证}
    F -->|不通过| C
    F --> G[提交代码人工Code Review]
    G -->|通过| H[任务完成]
    G -->|不通过| C
    style A fill:#e1f5fe
    style H fill:#c8e6c9

客户端开发的 checkpoint 设计：

阶段Checkpoint说明
1需求确认明确要做什麼
2规格定义OpenSpec 写清楚
3任务拆分&编排确定任务清单&上下游以来分析
4页面逻辑和自测用例编写AI基于TODO LIST进行开发和用例编写
5基于CLI的Lint验证按验收标准自测，保证无法没错
6人工Code Review开发者 review 功能是否正常
核心：
通过Trea的Rule固化工作流。不同周期调用不同的Skill。

关键实践：

  • 每次只做一个 checkpoint，减少上下文书积压
  • 编译失败立即停止，不盲目继续
  • 按验收标准自测，而非凭感觉

5. 总结

关键要点

  1. Harness 是护城河：模型是商品，差异来自 Harness 质量
  2. 六大组件：上下文工程 + 工具编排 + 状态管理 + 验证安全 + 人机协同 + 生命周期
  3. 三层规则：全局规则 + 项目规则 + Skills
  4. 验证优先：TDD 约束 Agent，防止跳过验证
  5. 渐进建设：从简单开始，观察失败模式再添加基础设施
